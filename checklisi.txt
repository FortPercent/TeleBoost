TeleBoost Wan2.1 T2V Pixel 加权迁移代码检查清单
===============================================

目标
----
请只 review 新增的 `_pixel` 分支实现，确认它是否把 DanceGRPO 参考脚本中的 pixel-weighted advantage 逻辑，正确迁移到 TeleBoost 当前 Wan2.1 T2V 训练链路中。

注意范围
--------
1. 不需要 review 原始主流程是否优雅，主流程没有被刻意修改。
2. 重点看 `_pixel` 新文件之间的数据流、shape、一致性和潜在运行时错误。
3. 不要求你运行训练；以静态代码审查为主。

参考来源
--------
参考逻辑来自：
/data/aigc/liangyzh_intern/zqni/DanceGRPO-main/fastvideo/train_grpo_wan_fsdp_pixel_wise_dino_on_adv.py

参考时重点对照的逻辑点：
1. DINO reverse semantic map 生成
2. reward + pixel map 组装
3. dense log-prob 路径
4. scalar advantage -> dense advantage

你需要回答的核心问题
--------------------
1. 这套 `_pixel` 链路是否能在不影响主流程的前提下独立工作？
2. pixel 权重图的计算、尺寸对齐、dtype/device 迁移是否正确？
3. rollout 输出的 dense `log_probs`、trainer 生成的 dense `advantages`、actor 消费的 dense loss 三者 shape 是否自洽？
4. 这套实现是否真正复现了参考脚本的核心算法，而不是只做了表面拼接？
5. 是否存在明显的 batch 维度、时间维度、latent 空间维度错位问题？
6. 是否存在潜在的 silent bug：广播错误、view/reshape 错误、detach 不当、device 不一致、pop 掉关键字段后下游再访问等？

总体验收标准
------------
[ ] 主流程入口未被破坏，只有 `_pixel` 入口才会启用新逻辑
[ ] `_pixel` 版本从入口 -> trainer -> worker -> rollout/actor 的引用链闭环正确
[ ] dense log-prob 的 shape 与 dense advantage 的 shape 在 loss 计算前严格兼容
[ ] pixel map 与 latent 时空分辨率对齐，没有隐式错位
[ ] reward 标量归一化与 dense 扩展逻辑和参考脚本一致或有充分理由解释差异
[ ] 新逻辑不会误伤 ref policy / KL / metrics 等依赖路径
[ ] 没有明显运行时崩溃点

请重点怀疑的高风险点
------------------
1. shape 风险
   - dense `log_prob`
   - dense `advantages`
   - `pixel_weight_maps`
   - actor loss reduction 前后的维度

2. 语义风险
   - 参考脚本是“标量 reward -> 标量 advantage 归一化 -> dense 加权”
   - 请确认当前迁移是不是这个顺序，而不是先 dense 后再做不一致归一化

3. ref/KL 风险
   - ref actor 也切成 pixel 版后，是否会影响 KL 路径原本默认的 shape 假设

4. 数据生命周期风险
   - rollout 写出的 `pixel_weight_maps` 是否在 trainer 中被安全消费
   - trainer `pop` 后是否还有任何后续逻辑会再访问它

5. 数值稳定性风险
   - PCA/SVD 近似
   - map 归一化
   - 全零/低方差 case
   - dense ratio / clip 的数值范围
